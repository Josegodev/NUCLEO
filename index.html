Problema

Las organizaciones estan adoptando LLM sin gobernar su operacion.

El problema no es solo la dependencia de un proveedor. Es la perdida de control sobre datos, fuentes, costes, permisos, trazabilidad, evaluacion y seguridad cuando el modelo se integra como una caja negra dentro de procesos reales.

Dependencia cerrada

Plataformas externas concentran acceso, ejecucion, datos y costes. La organizacion usa el modelo, pero no controla completamente el sistema operacional que lo rodea.

Baja trazabilidad

Sin trazas claras no se puede reconstruir que prompt, fuente, tool, politica o modelo produjo una respuesta o una accion.

Evaluacion debil

La calidad suele medirse por impresion. Faltan golden sets, metricas de evidencia, pruebas de seguridad y seguimiento de coste y latencia.

Hipotesis

Una arquitectura stateless, auditable y reproducible puede mejorar el control organizacional sobre el uso de LLM.

El valor no esta solo en el modelo. El valor esta en gobernar la operacion: politicas, permisos, fuentes, herramientas, logs, evaluacion y trazas.

Modelo de gobernanza

NUCLEO estructura el uso de LLM como un sistema gobernado, no como un chatbot.

01

Acceso

Identidad, rol, contexto y limites de consulta antes de recuperar informacion o invocar herramientas.

02

Fuentes

Respuestas apoyadas en documentacion verificable, con distincion explicita entre evidencia suficiente y ausencia de evidencia.

03

Politicas

Reglas de ejecucion y autorizacion separadas del modelo. El LLM puede proponer; la policy decide.

04

Herramientas

ToolRegistry como frontera de capacidades: solo se ejecutan tools registradas, contratadas y autorizadas.

05

Observabilidad

Logs, trace_id, estado de ejecucion, errores, decisiones de policy y contexto operativo por consulta.

06

Evaluacion

Golden set, ataques de prompt injection, fuga de datos, calidad de respuesta, coste, latencia y cobertura de fuentes.

Prototipo

NUCLEO es la implementacion experimental del modelo.

El repositorio implementa un runtime modular sobre FastAPI para separar decision, validacion y ejecucion. Su objetivo es probar que una organizacion puede operar LLM con mas control, menos opacidad y una base metodologica evaluable.

Componentes del prototipo

  • API de acceso y autenticacion por request
  • AgentRuntime como capa de orquestacion
  • Planner para propuestas controladas
  • PolicyEngine con comportamiento deny-by-default
  • ToolRegistry para herramientas disponibles
  • Approval gate para separar propuesta y ejecucion
  • Observabilidad y trazas internas
  • Evaluacion mediante casos y pruebas

Metricas

Lo que no se mide no se gobierna.

Estas metricas definen la evaluacion del caso de uso. No se publican valores inventados: el prototipo debe generar el baseline de forma reproducible.

Coste por consulta Control economico y comparacion entre proveedores o modelos.
Latencia Tiempo de respuesta por consulta, ruta y backend.
Tasa de respuestas con fuente Porcentaje de respuestas apoyadas en evidencia documental.
Fallos de politica Intentos bloqueados, denegaciones y acciones fuera de contrato.
Resistencia a prompt injection Pruebas adversariales contra instrucciones maliciosas o contradictorias.
Fuga de datos Deteccion de exposicion no autorizada de informacion sensible.
Calidad en golden set Evaluacion recurrente sobre preguntas y respuestas esperadas.

Resultado esperado

Marco metodologico

Un esquema operativo para evaluar y gobernar LLM en organizaciones, separando acceso, fuentes, politicas, tools y evaluacion.

Prototipo reproducible

Una base FastAPI verificable que demuestra el pipeline de control y permite evolucionar hacia casos documentales reales.

Caso evaluado

Un asistente documental para entorno universitario, LMS o empresa con metricas tecnicas, operativas y de seguridad.

Estado actual: prototipo avanzado y experimental. No debe presentarse como producto de produccion ni como garantia de cumplimiento normativo. Su contribucion es metodologica y tecnica: convertir el uso de LLM en una operacion observable, reproducible y evaluable.
