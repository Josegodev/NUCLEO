# Controlled LLM Planner Augmentation

## Current State

NUCLEO has a controlled LLM augmentation boundary inside the Planner. The LLM
may propose a structured action, but it cannot authorize, resolve, or execute a
tool.

The production execution order remains:

```text
AgentRequest
-> AgentService
-> AgentRuntime
-> Planner
-> PolicyEngine
-> ToolRegistry
-> Tool or dry-run proposal
-> AgentResponse
```

For Productive Agent Console v0, the request uses:

```json
{
  "options": {
    "agent_mode": "proposal_only",
    "dry_run": true
  }
}
```

`proposal_only` allows LLM-assisted planning. `dry_run=true` prevents
`tool.run(...)`.

## Model Router

`app/adapters/model_router.py` is the runtime-facing model boundary.

Responsibilities:

- select `local`, `openai`, or `auto`
- call local Ollama through `runtime_lab/llm_lab/model_adapter.py`
- call OpenAI through the OpenAI chat completions HTTP API
- return one normalized result contract

Normalized result:

```json
{
  "output": "...",
  "model_used": "...",
  "backend_used": "...",
  "latency_ms": 123.0,
  "fallback_used": false,
  "fallback_reason": null
}
```

The runtime does not call `runtime_lab/llm_lab` directly. It calls the
`ModelRouterProposalProvider`, which calls `ModelRouter`.

## Prompt Contract

`ModelRouterProposalProvider` builds a prompt with:

- the user goal
- structured context
- fixed safety constraints
- the active tool contract catalog

The catalog is generated by:

```text
build_tool_contract_prompt(tool_registry)
```

That function reads `tool_registry.list_contracts()` and renders the registered
tool names plus exact argument schemas. This prevents hardcoding tool-specific
prompt patches such as `message -> text`.

Example catalog fragment:

```text
Available tools and required argument schemas:

- echo
  arguments:
    text: string

- system_info
  arguments:
    {}
```

Rules sent to the model:

- use only listed tools
- use exact argument names
- do not invent fields
- do not use aliases
- if no tool fits, return `suggested_action=null` and `arguments={}`
- output only JSON, optionally wrapped in a `json` fenced block

## Output Contract

Accepted LLM output shape:

```json
{
  "intent": "echo request",
  "suggested_action": "echo",
  "arguments": {
    "text": "hola"
  },
  "confidence": 0.9
}
```

`suggested_action` may be `null` only when no action should be planned. In that
case `arguments` must be `{}`.

Free text is invalid. Extra payload fields are invalid because payloads are
validated against the selected tool Pydantic contract.

## JSON Normalization

`_strip_json_fence(raw)` accepts only:

- pure JSON
- a full Markdown fenced block whose opening fence is empty or labelled `json`
  and whose final line is a closing fence

The normalizer does not extract JSON from narrative text. These are invalid:

- text before or after JSON
- an opening fence without closing fence
- fenced languages other than empty or `json`

The normalized string is passed to `json.loads(...)`.

## Validation

`LLMPlanOutputValidator` performs the runtime-side checks:

1. parse JSON
2. validate required fields
3. validate `confidence` range
4. accept `suggested_action=null` only with empty arguments
5. validate tool name with the closed tool-name set
6. verify the tool exists in the active `ToolRegistry`
7. validate `arguments` with the tool payload schema
8. convert the accepted proposal to `PlannedAction(source="llm_assisted")`

The accepted proposal is still not authorization. Runtime sends the
`PlannedAction` through `PolicyEngine`, then `ToolRegistry`, then payload
validation before dry-run or execution.

## Fallback

`LLMAssistedPlannerStrategy` falls back to deterministic planning when:

- the selected model backend fails
- the model output is empty
- JSON normalization or parsing fails
- the proposal shape is invalid
- the tool is unknown or unregistered
- the payload does not match the selected tool contract

Fallback metadata is attached to the resulting plan when available:

- `augmentation_attempted`
- `backend`
- `model_id`
- `backend_used`
- `model_used`
- `latency_ms`
- `fallback_used`
- `fallback_reason`

If deterministic planning also returns `no_plan`, the response may be rejected,
but it still carries metadata showing that augmentation was attempted.

## Approval Boundary

Planner augmentation stops at a proposal. Real execution is controlled by the
Approval Gate:

```text
PROPOSED -> APPROVED -> EXECUTED
PROPOSED -> REJECTED
```

`POST /agent/approve` does not call `ModelRouter`, does not call
`planner_augmentation`, and does not call the Planner. It loads the persisted
proposal by `trace_id`, revalidates `PolicyEngine`, resolves `ToolRegistry`
again, validates the persisted payload, and only then calls `tool.run(...)`.

## Invariants

- LLM output is never executed directly.
- LLM output never creates a `PolicyDecision`.
- LLM output never mutates `ToolRegistry`.
- The frontend cannot edit executable payloads during approval.
- `PolicyDecisionValue` remains `ALLOW | DENY`.
- `dry_run` remains a runtime execution flag.
- Multi-step planning is not implemented.
- Conversational memory is not implemented.
